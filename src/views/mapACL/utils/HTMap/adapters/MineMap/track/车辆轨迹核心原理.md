# 车辆轨迹（Track）核心原理文档

## 一、概述

车辆轨迹模块（TrackManager）用于在地图上展示**轨迹线**和**车辆符号**，并支持**实时追加点位、更新轨迹**。实现方式与 MineData 示例一致：**轨迹线**由 LinesManager 绘制，**车辆图标**由 Mapbox 风格的 Symbol Layer（点 GeoJSON + `setData` 实时更新）绘制。

### 核心组成

| 组成部分     | 技术实现                          | 作用           |
|--------------|-----------------------------------|----------------|
| 轨迹线       | LinesManager（LineString 几何）   | 显示历史路径   |
| 车辆符号     | GeoJSON Source + Symbol Layer     | 显示当前位置与朝向 |
| 航向角计算   | 两点间方位角公式                  | 车头随行驶方向旋转 |

---

## 二、架构与数据流

### 2.1 类职责

```
TrackManager
├── 轨迹存储：Map<id, track>
├── 轨迹线：委托 linesManager.addLines / updateLinesGeometries
├── 车辆图标：map.addSource(geojson) + map.addLayer(symbol)
└── 图标资源：单例加载车辆图片，addImage 注册
```

- **轨迹线**：每条轨迹对应一个 lineGroup（LinesManager 返回），内含一条 LineString，路径点数组随 `updateTrack` / `pushTrackPoint` 更新。
- **车辆符号**：每条轨迹对应一个 GeoJSON Source（单点 Feature）和一个 Symbol Layer；更新位置/朝向时只调用 `source.setData(newFeature)`，不增删 Layer。

### 2.2 单条轨迹内部结构

每条轨迹在内存中保存为：

```js
{
  id,                    // 轨迹唯一 ID
  path,                  // 当前路径点数组 [[lng, lat], ...]（已坐标转换）
  lineGroup,             // LinesManager 返回的线段组，用于更新/删除线
  symbolSourceId,        // 车辆点 GeoJSON Source 的 ID
  symbolLayerId,        // 车辆 Symbol Layer 的 ID
  geomId,                // 线段几何 ID（更新线时用）
  styleId,               // 线段样式 ID
  options: { ... }       // 创建时的配置快照（含 lineStyle、symbol、label 等）
}
```

- **path**：始终为“已转换坐标”的数组，供画线和计算车头朝向使用。
- **lineGroup**：持有 `updateLinesGeometries`、`removeLines`，用于刷新线段几何或移除线段。
- **symbolSourceId / symbolLayerId**：用于更新车辆点或移除图层/数据源。

## 三、核心算法与公式

### 3.1 航向角（Bearing）计算

车辆图标需要随行驶方向旋转，即计算**从上一点到当前点的方位角**（0° 为正北，顺时针增加）。

```js
function getBearing(from, to) {
  const [lng1, lat1] = from
  const [lng2, lat2] = to
  const dLon = (lng2 - lng1) * Math.PI / 180
  const lat1Rad = lat1 * Math.PI / 180
  const lat2Rad = lat2 * Math.PI / 180
  const y = Math.sin(dLon) * Math.cos(lat2Rad)
  const x = Math.cos(lat1Rad) * Math.sin(lat2Rad) - Math.sin(lat1Rad) * Math.cos(lat2Rad) * Math.cos(dLon)
  return (Math.atan2(y, x) * 180 / Math.PI + 360) % 360
}
```

- 公式本质：球面三角中的**方位角公式**，将两点经纬度差转为弧度，用 `atan2(y, x)` 得到角度，再归一化到 `[0, 360)`。
- 单点轨迹（path 长度为 1）时无“上一段”，朝向默认为 0。

### 3.2 图标旋转与偏移

地图引擎中图标通常“默认朝上”，而方位角 0° 表示正北，因此需要**偏移**使车头与行驶方向一致：

```js
function iconRotation(travelBearing, offset = 0) {
  return (travelBearing + offset + 360) % 360
}
```

- **travelBearing**：由 `getBearing` 得到的行驶方向（度）。
- **offset**：默认 `-90`（DEFAULT_BEARING_OFFSET），即把“北”对齐到“向右”，使 0° 对应车头朝东等；不同图标可配置不同 `symbol.bearingOffset`。

Symbol Layer 中通过 `layout['icon-rotate']: ['get', 'bearing']` 读取 Feature 的 `properties.bearing`，因此每个点 Feature 需带上计算好的 `bearing`。

### 3.3 坐标转换

所有输入坐标在写入 `path` 或传给线几何前，都经 `transformCoordinates(coordinates, context)` 处理：

- 保证格式为 `[lng, lat]` 且为数字；
- MineMap 适配器下不做坐标系变换，仅校验与标准化，便于未来扩展为其他坐标系（如 GCJ02/WGS84 等）。

## 四、车辆点 GeoJSON 与 Symbol 配置

### 4.1 点 Feature 结构

车辆当前拉置用**单个 Point Feature** 表示：

```js
function pointFeature(pos, bearing, plate = '') {
  return {
    type: 'Feature',
    geometry: { type: 'Point', coordinates: pos },
    properties: { bearing, plate: plate || '' }
  }
}
```

- **pos**：当前点 `[lng, lat]`（已转换）。
- **bearing**：图标旋转角（度），即 `iconRotation(travelBearing, bearingOffset)` 的结果。
- **plate**：可选标签（车牌等），用于 Symbol 的 `text-field`。

### 4.2 Symbol Layer 配置要点

- **icon-image**：使用统一注册的 `TRACK_VEHICLE_ICON_ID`（如 `'track-vehicle-icon'`），图片通过 `_ensureVehicleIcon()` 预加载并 `map.addImage()` 注册。
- **icon-size**：默认 0.06，可由 `options.symbol.size` 覆盖。
- **icon-rotate**：`['get', 'bearing']`，使图标随 `properties.bearing` 旋转。
- **icon-rotation-alignment: 'map'**：旋转相对于地图北方向，适合车辆等方向性图标。
- **text-field**：`['get', 'plate']`，显示车牌/标签；`text-optional: true` 表示无文字时不报错。
- **text-anchor / text-offset**：控制文字相对图标的位置（如底部偏上）。

这样，**只更新 GeoJSON Source 的 data**（调用 `setData(pointFeature(...))`），即可实现车辆位置与车头朝向的实时更新，无需增删 Layer。

## 五、轨迹线（LinesManager）协作

### 5.1 添加轨迹时创建线段

- 使用 `linesManager.addLines()` 创建一条线：
  - **geometries**：一条几何，`paths` 为当前 `path`（至少 2 点；若只有 1 点则用 `[point, point]` 避免无效线）。
  - **styles**：一条样式，指定 color、width、opacity、lineJoin、lineCap、dashArray 等。
- 返回的 `lineGroup` 保存到 `track.lineGroup`，并记录 `geomId`、`styleId` 供后续更新。

### 5.2 更新轨迹时刷新线段

- 在 `updateTrack(id, path)` 中，用新的 `transformedPath` 更新内存中的 `track.path`。
- 若存在 `track.lineGroup.updateLinesGeometries`，则调用：
  - `updateLinesGeometries([{ id: track.geomId, paths: lineCoords, properties: { trackId: id }, styleId: track.styleId }])`
- **lineCoords**：同样在只有 1 点时用 `[point, point]`，保证 LineString 有效。

这样轨迹线始终与 `path` 一致，实现“历史路径”的实时延伸。

## 六、主要 API 行为说明

### 6.1 addTrack(options)

- **前置**：`mapInstance.map` 存在，`options.id` 存在，`options.path` 为非空数组。
- **重复 id**：若已存在同 id 轨迹，先 `removeTrack(id)` 再创建。
- **流程**：
  1. 对 path 逐点 `transformCoordinates` 得到 `transformedPath`。
  2. 用 LinesManager 添加轨迹线，保存 `lineGroup`、`geomId`、`styleId`。
  3. 异步 `_ensureVehicleIcon()`，完成后：
     - 取当前 path 末点与倒数第二点算 `travelBearing`，算 `bearing`，构造 `pointFeature`；
     - 若无对应 Source 则 `addSource(geojson)`，若有则 `setData(...)`；
     - 若无对应 Symbol Layer 则 `addLayer(symbol 配置)`；
     - 将 `symbolSourceId`、`symbolLayerId` 写回 track（因为先存了 track 再异步写回）。
  4. 将 `{ id, path, lineGroup, symbolSourceId, symbolLayerId, geomId, styleId, options }` 存入 `this.tracks`。
- **options 常用字段**：
  - **id**：必填，唯一标识。
  - **path**：必填，`[[lng, lat], ...]`。
  - **showLine**：是否显示轨迹线，默认 `true`；为 `false` 时不创建线段。
  - **showPlate**：是否显示车牌（Symbol 的 text-field），默认 `true`；为 `false` 时不显示文字。
  - **showSpeed**：是否显示速度（车牌下一行），默认 `true`；为 `false` 时不显示速度行。显示时速度会自动带单位 `km/h`（若传入值已含 `km/h` 则不再重复添加）。
  - **lineStyle**：`{ color, width }` 等。
  - **symbol**：`{ size, bearingOffset }`。
  - **label / plate**：车辆标签文字（车牌，显示在第一行）。
  - **speed**：车速（可传数字如 `60` 或带单位如 `"60 km/h"`），显示在车牌下一行时会自动补单位 `km/h`；可通过 `updateTrack(id, path, { speed })` 或 `pushTrackPoint(id, lngLat, { speed })` 动态更新。

### 6.2 pushTrackPoint(id, lngLat, extraOptions)

- 将一点追加到指定轨迹的 path 末尾。
- **extraOptions**（可选）：如 `{ speed: '60 km/h' }`，会合并到 `track.options` 并随本次更新刷新符号文案（如车速）。
- 内部：`coord = 标准化为 [lng, lat]` → `updateTrack(id, track.path.concat([...]), extraOptions)`。

### 6.3 updateTrack(id, path, extraOptions)

- 用新的 path（数组）**整体替换**该轨迹的路径。
- **extraOptions**（可选）：如 `{ speed: '60 km/h' }`，会合并到 `track.options`，并用于本次 Symbol 的 setData（如车牌下一行车速）。
- **流程**：
  1. 对 path 逐点 `transformCoordinates`，更新 `track.path`。
  2. 取末点与倒数第二点算 bearing，用 `pointFeature` 更新 `track.symbolSourceId` 的 `setData`。
  3. 若有 `lineGroup.updateLinesGeometries`，用当前 `track.path` 更新线几何（仍处理单点情况为 `[p, p]`）。

因此无论是“追加一点”还是“整段重算”，都统一用 `updateTrack` 保证线和点数据一致。

### 6.4 removeTrack(id)

- 移除该轨迹的 Symbol Layer、GeoJSON Source、线段（lineGroup.removeLines），并从 `this.tracks` 中删除。

### 6.5 getTrack(id) / getAllTracks() / removeTracks()

- **getTrack(id)**：返回 `this.tracks.get(id)`。
- **getAllTracks()**：返回所有 track 的数组。
- **removeTracks()**：对每个 id 调用 `removeTrack(id)`，停止并移除所有轨迹。

## 七、图标加载与复用

- **单例加载**：车辆图标通过 `_ensureVehicleIcon()` 只加载一次，用 `_iconPromise` 缓存 Promise；若已 `map.hasImage(TRACK_VEHICLE_ICON_ID)` 则直接 resolve。
- **加载流程**：`new Image()` 加载默认车辆图（如 `car.png`），`onload` 里 `map.addImage(TRACK_VEHICLE_ICON_ID, img, { pixelRatio: 2 })`，所有轨迹的 Symbol Layer 共用该 image。
- **时机**：在 `addTrack` 里异步调用 `_ensureVehicleIcon().then(...)` 再创建 Source/Layer，避免图未加载完就渲染。

## 八、与业务层配合（示例：mapVehicleTrack）

- **多车多路线**：每条轨迹一个 id，path 可由“路线”生成（如东西向/南北向/环线），业务层维护 `vehiclePlates`、`vehicleRoute` 等。
- **模拟行驶**：定时器每帧对每辆车取当前路线的下一索引点，调用 `map.pushTrackPoint(id, path[idx])`，再 `updateStats()` 同步列表；实现“多路线路口车流”循环行驶。
- **引擎支持**：仅当引擎为腾讯地图或 MineMap 时暴露轨迹能力（依赖 LinesManager + GeoJSON/Symbol），其他引擎可提示不支持或走兼容实现。

## 九、小结

| 要点           | 说明 |
|----------------|------|
| 双图层         | 轨迹线（LinesManager）+ 车辆点（GeoJSON Symbol），各司其职。 |
| 实时更新       | 线：`lineGroup.updateLinesGeometries`；点：`source.setData(pointFeature)`。 |
| 航向角         | 由路径末两点用球面方位角公式计算，加 bearingOffset 后写入 Feature。 |
| 坐标统一       | 所有写入 path/线的坐标均经 `transformCoordinates`。 |
| 单点路径       | 线几何至少 2 点，单点时用 `[point, point]` 避免无效 LineString。 |
| 资源复用       | 车辆图标全局加载一次，多轨迹共用同一 image id。 |

掌握以上原理后，即可在扩展其它地图引擎或改造样式/交互时保持轨迹行为一致，并便于排查“线不更新、车不转、坐标错”等问题。
